# PR / コミットメッセージ / 組織共有ドキュメントは共有コンテキストで書く

PR title / body / commit message / PR レビューコメント（approve・change request 含む）、およびリポ内 `docs/*.md` や Kibela 周知記事など組織共有ドキュメントには、チーム外から参照できない個人固有の語彙を書かない。

「組織共有ドキュメント」は具体的に以下を含む。

- PR / コミット / レビューコメント（個人 → チームへ）
- リポ内 `docs/*.md` `README.md` `CONTRIBUTING.md` `ROLLOUT.md` 等（個人 → 同リポ利用者へ）
- Kibela / チーム Wiki エントリ（個人 → 全社へ）
- Slack の告知投稿（個人 → channel メンバーへ）

## 避ける表現

- 個人 skill 名（例: `/harness-audit`, `/post-merge`, `/work-log`, `/review-pr` など、チームに共有されていない自作スキル）
- 個人 skill 内のカテゴリ名（例: `should-fix`, `must-fix`, `P0/P1`, `must-fix なし` など、内部レビューツールの出力フォーマット由来の語彙）
- Codex 等、個人ローカル環境で使う AI セカンドオピニオンツールへの言及（例: 「Codex second opinion」「Codex 提案」「Codex 確認済み」）。チームメンバーには参照不能なため、結論を裏付ける技術的事実（公式 docs・実観測ログ等）だけを書く
- 個人 dotfiles / ローカルノートのパス（例: `~/dotfiles-private/...`、Obsidian デイリーノートの相対リンク）
- チームのリポジトリに存在しないローカル運用スクリプト名
- 実装者の作業状態（`stash 済`、`既に退避済` 等）— 読む側には不要な情報
- 内部検討の名前（`案X`、`段階2` 等）— チケット・会話の外では意味が通じない
- フェーズ管理の用語（`Phase 1 POC 完了済` 等）— 前提知識がないと理解できない

## 理由

レビュワー / PR 作成者にはこれらのリソースが共有されておらず参照不能。PR の背景が追えず、レビュー効率が落ちる。

PR #6925・#6931 で PR 本文に内部用語を残し、PR #6977 の approve コメントで「review-pr で should-fix 2件」という内部表現を使ってしまい、PR 作成者から見て意味不明な表現になった。2026-07-08 には PR #65 (PF-2793) の PR 本文・Linear Issue 本文で「Codex second opinion 確認済み」と書いて同様の指摘を受けた。

## 書き換え方

作業着手のトリガーが個人スキル起動でも、PR 本文・レビューコメントにはその事実を書かない。レビューツールやスキル名を一切知らない第三者の視点で読み返す。

内部レビュープロセスの詳細はユーザーへの報告にとどめ、PR 上には「何を / どう確認したか」「なぜ approve / 修正依頼か」を技術的な事実だけで書く（例: 「`destroy = false` で実 IAM 削除は起きないため許容」「3 パターンで判定結果が意図通り出ることを確認」）。

言及している全リソースが org / チームから参照可能かを確認し、参照不能なら以下のいずれかに翻訳する。

- org リポに既に存在する先行事例（`<org>/<repo>` の該当ファイル）
- Kibela / チーム Wiki エントリ
- リポジトリ内の ADR（`docs/adr/`）
- Linear Issue / Slack permalink

## 読み手環境を前提にしない（PM 向け / 全社向けドキュメント）

社内利用者向けのドキュメント（リポ内 docs / Kibela 周知記事等）を書くときは、読み手の環境が個人と同じであることを前提にしない。

- 個人ツール前提を避ける: 例 `ghq get` ではなく一般的な `git clone` を使う。`direnv`・`bw-fetch` 等の個人 dotfiles ツールを手順に組み込む前に、社内で標準化されているか確認する
- リンクは転載先で動く形に: リポ内ドキュメントに相対パス（`./xxx.md`, `../db/yyy.md`）を書くと、Kibela に転載した際にリンク切れする。組織共有を前提とする `ROLLOUT.md` `ONBOARDING.md` 等は GitHub 絶対 URL（`https://github.com/<org>/<repo>/blob/main/<path>`）で書いておくと、リポ内 UI と Kibela 転載先の両方で動作する
- セットアップ前提を明示: 「Claude Code 利用可」「Bitwarden CLI セットアップ済」のような前提条件は冒頭に列挙する。前提を満たさない読み手が無限ループする事故を防ぐ
- フォルダ構造の listing は深掘りする: Drive / Linear / ファイルシステムなど階層のあるリソースを参照する手順を書くときは、直下だけ見ずに「提出済みファイル/」「アーカイブ/」「YYYY年/」等のサブフォルダにも目を通す。書き手側で実体を確認してから手順を作る
