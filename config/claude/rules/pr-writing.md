# PR / コミットメッセージ / レビューコメントは共有コンテキストで書く

PR title / body / commit message / PR レビューコメント（approve・change request 含む）には、チーム外から参照できない個人固有の語彙を書かない。

## 避ける表現

- 個人 skill 名（例: `/harness-audit`, `/post-merge`, `/work-log`, `/review-pr` など、チームに共有されていない自作スキル）
- 個人 skill 内のカテゴリ名（例: `should-fix`, `must-fix`, `P0/P1`, `must-fix なし` など、内部レビューツールの出力フォーマット由来の語彙）
- 個人 dotfiles / ローカルノートのパス（例: `~/dotfiles-private/...`、Obsidian デイリーノートの相対リンク）
- チームのリポジトリに存在しないローカル運用スクリプト名
- 実装者の作業状態（`stash 済`、`既に退避済` 等）— 読む側には不要な情報
- 内部検討の名前（`案X`、`段階2` 等）— チケット・会話の外では意味が通じない
- フェーズ管理の用語（`Phase 1 POC 完了済` 等）— 前提知識がないと理解できない

## 理由

レビュワー / PR 作成者にはこれらのリソースが共有されておらず参照不能。PR の背景が追えず、レビュー効率が落ちる。

PR #6925・#6931 で PR 本文に内部用語を残し、PR #6977 の approve コメントで「review-pr で should-fix 2件」という内部表現を使ってしまい、PR 作成者から見て意味不明な表現になった。

## 書き換え方

作業着手のトリガーが個人スキル起動でも、PR 本文・レビューコメントにはその事実を書かない。レビューツールやスキル名を一切知らない第三者の視点で読み返す。

内部レビュープロセスの詳細はユーザーへの報告にとどめ、PR 上には「何を / どう確認したか」「なぜ approve / 修正依頼か」を技術的な事実だけで書く（例: 「`destroy = false` で実 IAM 削除は起きないため許容」「3 パターンで判定結果が意図通り出ることを確認」）。

言及している全リソースが org / チームから参照可能かを確認し、参照不能なら以下のいずれかに翻訳する。

- org リポに既に存在する先行事例（`<org>/<repo>` の該当ファイル）
- Kibela / チーム Wiki エントリ
- リポジトリ内の ADR（`docs/adr/`）
- Linear Issue / Slack permalink
