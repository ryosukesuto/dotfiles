# Subagentのモデル使い分け

メイン会話は既定モデルのまま。subagent起動時の`model:`パラメータで明示的に選ぶ。

## 判断マトリクス

| モデル | 用途 | 具体例 |
|-------|------|-------|
| `haiku` | 定型・機械的な処理 | 文字列整形、grep結果のフィルタ、ログ抽出、簡単なファイル名列挙 |
| `sonnet`（既定） | 通常の実装・調査 | Explore（quick/medium）、Plan、一般的な実装、code review |
| `opus` | 深い思考が要るもの | 設計判断、難バグの根本原因調査、アーキテクチャ検討、大規模リファクタ計画 |

## Haikuを指定する条件

以下を全て満たすときに`model: "haiku"`を指定する。

- 出力形式が決まっている（list化・整形・抽出）
- 判断要素が少なく、正誤が明確
- 失敗しても影響が小さい（読み取り専用、試行錯誤可能）

典型例:
- Exploreでquickモード、かつキーワード検索だけで済むとき
- grep/lsの結果から特定パターンを抽出するとき
- ファイル一覧を整形して返すとき

## Haikuを避けるケース

- 設計意図の推測が必要（`why`を問う系）
- 複数ファイルを横断して判断する調査
- エラーの原因特定や修正提案

迷ったらSonnetを使う。Haikuで品質が足りない場合の切り戻しコストのほうが高い。

## Opusを指定する条件

- 設計判断を伴うPlan agent
- アーキテクチャ検討や難バグ調査
- 複数の選択肢からトレードオフを評価する場面

コストが高いため、Sonnetで足りるかをまず検討する。

## 指定方法

```
Agent({
  subagent_type: "Explore",
  model: "haiku",
  description: "...",
  prompt: "..."
})
```

`model:`省略時はsubagent定義のfrontmatter、または親の既定モデルを継承する。
