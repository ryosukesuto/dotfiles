# update-security-db リファレンス

## 事実層のジャンル一覧

事実層は以下のジャンルで構成される。新情報はいずれかのジャンルに分類して追記する。

| ジャンル | 内容 |
|---------|------|
| 会社概要・組織体制 | 運営会社、セキュリティ組織、評価フレームワーク |
| コンプライアンス・法令遵守 | 法令対応、反社チェック、内部通報 |
| 本人確認（KYC） | eKYC、公的書類、顔認証 |
| 認証・アクセス制御 | ユーザー認証、管理者認証、コード管理 |
| 取引認証 | 入金・出金時の認証方法 |
| 取引/決済モニタリング | 不正取引検知、ML、投票モニタリング |
| 不正取引に関する補償 | 補償規定、利用規約 |
| 不正アクセス対策・ネットワークセキュリティ | WAF、DDoS、レート制限、Zero Trust |
| 脆弱性管理 | 診断、スキャン、依存関係管理 |
| 不正検知 | レート制限、ML検知、位置偽装検知 |
| 暗号化・通信保護 | TLS、mTLS、KMS、鍵ローテーション |
| ログ管理・監査 | 監査ログ、監視、トレーシング |
| インシデント対応 | エスカレーション、ポストモーテム、訓練 |
| インフラ・クラウド | GCP、AWS、CDN、IaC |
| 開発セキュリティ（SDLC） | リスク評価、コードレビュー、スキャン |
| 情報漏洩対策 | エンドポイント、暗号化、PC廃棄 |
| 外部委託管理 | クラウド事業者、認証確認 |
| サービス仕様 | チャージ、精算、利用規約 |
| 物理セキュリティ | オフィス（親会社管理） |

## 事実層の記述ルール

### 書いてよい表現

- 一般的なセキュリティ用語: WAF、MFA、TLS、IDS/IPS、SIEM、DDoS、mTLS、IAM、KMS、eKYC
- 対策の種類と方法: 「WAFを導入し、SQLインジェクション・XSS等を検知・遮断」
- 運用頻度: 「年1回以上」「毎月」「24時間365日」
- クラウド事業者名: GCP、AWS（公開情報のため）

### 書いてはいけない表現

- 内部ツール固有名: Wiz, Vertex AI, Casbin, Terraform, ArgoCD, Datadog, Sentry, Fastly, Cloudflare
- 具体的な閾値: 「30リクエスト/10秒」等のレート制限値
- 内部組織名: SSG, PFチーム等（「セキュリティ専門部署」「専任チーム」と表現）
- 内部URL: Kibela, Linear, Slackチャンネル名
- 内部フレームワーク名: Security CREST（「NIST CSF 2.0ベースの評価」と表現）

### 書き方の例

悪い例: Wizを導入し、Cloud Armorのレート制限を30req/10秒に設定
良い例: クラウドセキュリティ監視ツールを導入し、WAFのレート制限を適用

## 回答層の古い表現パターン

以下のパターンに該当する回答は更新候補。ただし文脈を確認してから判断する。

| パターン | 判断基準 |
|---------|---------|
| `検討中` `検討して` | 1年以上前の回答で「検討中」のままなら更新 |
| `予定です` `予定で` | 期日が過ぎている場合は実施状況を確認 |
| `確認中` | 長期間「確認中」のままなら具体的な回答に更新 |
| `未導入` `未実装` `未対応` | 最新情報で導入済みなら更新 |
| `20XX年度` | 過去の年度を参照している場合は更新 |

## 銀行別の回答傾向

回答を更新する際は、銀行ごとの傾向に合わせる。

| 銀行 | 傾向 | 対応方針 |
|------|------|---------|
| ゆうちょ | 深掘り質問が多い | 最初から詳細な補足を含めた回答に |
| みずほ | YES/NOベース | 備考で補足する程度 |
| SMBC | YES/NOベース | シンプルな回答 |
| SBI | チェックリスト形式 | セクション別に整理 |
| PayPay | アンケート形式 | はい/いいえの簡潔な回答 |

## ragent検索のコツ

- 短いキーワードより自然文の方が精度が高い
- `include_metadata: true` でメタデータを取得すると更新日が分かる
- 検索結果が大きい場合はAgentに委任して要約させる
- Slack検索は `enable_slack_search: true` で有効化可能（通常は不要）
