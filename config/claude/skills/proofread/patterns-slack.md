## Slack向け追加パターン

`/proofread slack` で起動したときに、共通patternsに追加で適用する。Slackは会話の場であり、ドキュメント体（見出し・箇条書き・体言止めの羅列）はかえって読みにくい。

### S1: 構造の脱ドキュメント化

| パターン | 修正方針 | 例 |
|---------|---------|-----|
| バッククォートで囲んだラベル + 改行（`` `原因` \n 〜 ``） | 1段落に統合し、接続詞で繋ぐ | `` `原因`\nlockfile が古い `` → 「原因としてはlockfileが古いといった感じで」 |
| ラベル + コロン（`原因:` `対応:`） | 同上。会話文に流す | 「原因: 〜」→「原因としては〜」 |
| `## 概要` `## 背景` などの Markdown見出し | 削除し、1文目で趣旨を要約 | 「## 概要\n〜」→ 1段落の文章に |
| 3項目以上の箇条書き | 文中で「〜と〜と〜」で並列、または重要な項目だけ残す | `- A\n- B\n- C` → 「AとBとCを実施」 |
| 体言止めの羅列（「〜が必要。」「〜済み。」） | 「〜が必要なんですが」「〜済みです」と接続詞で繋ぐ | 「対応必要。権限ある方お願い。」→「対応必要なんですが、権限ある方お願いできますか？」 |

### S2: トーン

| パターン | 修正方針 |
|---------|---------|
| 「〜お願いします。」断定調 | 「〜お願いできますか？」「〜お願いしてもよいですか？」（疑問形＋柔らかさ） |
| 「〜が必要です。」「〜してください。」指示調 | 「〜が必要なんですが」「〜していただけると」（相談調） |
| 「ご対応のほどよろしくお願いいたします。」過剰な敬語 | 「対応お願いできますか？」程度の温度に下げる |
| 「お疲れさまです。」冒頭の決まり文句 | 削除して本題から入る（チャンネルではすでに会話が流れているため） |
| 「〜です。」「〜ます。」だけが連続 | 「〜なんですが」「〜していて」を混ぜて自然な抑揚にする |

### S3: リンク・記号

| パターン | 修正方針 |
|---------|---------|
| URLを `()` で囲む（Markdown的） | 生URLを本文中に埋め込む。SlackのUnfurlで自動展開される | 
| `[テキスト](URL)` 形式 | `テキスト URL` または生URLのみ | 
| `**強調**` の Markdown 太字 | 削除する（Slackで `*` 1個になり違う意味になる、もしくは効きが弱い） | 
| 絵文字を装飾的に多用 | 必要なら最小限。原則ゼロでよい | 
| メンションが文末・段落末 | 文頭で `@user` をつけて誰宛か即座にわかるようにする | 

### S4: 結びの形

| パターン | 修正方針 |
|---------|---------|
| 「以上、よろしくお願いします。」型の締め | 削除。本文最後の依頼文で締まっていればOK |
| 「ご確認ください。」ふんわり依頼 | 「〜できますか？」「〜お願いできますか？」と具体的な動作を聞く |
| 末尾に補足ラベル（`参考:` `備考:`） | 自然な文に組み込むか削除 |

### 判断基準

- Slackは流し読みされる。1スクロール以内に収まる長さを目標にする
- 会話の中の発話として読めるかをチェックする。読めなければドキュメント体に寄りすぎ
- 共通patternsの「ガイドライン・手順書で〜する。の文末を体言止めに」の指針はSlackでは無効。Slackでは「〜なんですが」「〜していて」の口語接続を優先する
