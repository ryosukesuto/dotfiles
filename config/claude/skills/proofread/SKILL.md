# /proofread - AI生成テキストの校正

直前に書いた、または指定されたファイルのテキストについて、読み手に必要な理解が成立するかを確認してからユーザーのトーンに合わせて校正する。表面的なAIらしさだけを消して、根拠や責任の空白を隠さない。

## 実行手順

### 1. 対象テキストの特定

引数の解釈:
- ファイルパス: そのファイルを対象にする
- `slack`（または `slack` を含む文字列）: 直前に書いた / 提示したSlack草案テキストを対象にし、Slack向けパターンを併用する
- `pr-review`（またはPRレビューコメント・Approve・Request changes・インラインコメントの草案）: PRレビュー向けパターンを併用する
- 省略時: 直前にEdit/Writeしたファイル、または直前のメッセージで提示したテキスト

### 2. 意味と共有前提の確認

Claude Codeでは`~/.claude/rules/writing-grounding.md`、Codexでは`~/.codex/claude-rules/writing-grounding.md`を読み、文体修正より先に次を確認する。

- 想定読者と、その読者が参照できる情報
- 読後に理解・判断・実行できるようにしたいこと
- 事実、推測、提案、決定、未確認の区別
- 読み手が補完しないと成立しない因果関係や未定義語
- 状況を更新せず、文書の構造だけを進める文

不足があれば、語彙や語尾より先に内容を補正する。根拠を確認できない情報は補完しない。

元文の情報だけでは読み手に必要な理解が成立しない場合は、流暢な文章にして完了扱いしない。中心的な主張、決定状態、読み手へ求める行動、重要な事実関係のいずれかが矛盾・欠落している場合はcriticalな不足として扱い、校正案を作る前に最小限の質問を返して停止する。

criticalではない根拠不足や未定義語は、確認できた事実だけで校正案を作り、校正案の外に「確認が必要な点」として分ける。曖昧な予定を決定済みにする、矛盾する記述から中間状態を作る、曖昧な依頼へ具体的な行動を創作するなど、元文にない状態や意図を補わない。

### 3. チェックリストに沿って検出

対象テキストを読み、以下のパターンを検出する。該当箇所を行番号付きでリストアップする。
自分が書いた文章のパターンは自分では見落としやすいため、チェックリストで機械的に洗い出す。

検出パターン:
- 共通: `${CLAUDE_SKILL_DIR}/patterns.md`
- Slack向け（引数 `slack` のときのみ追加適用）: `${CLAUDE_SKILL_DIR}/patterns-slack.md`
- PRレビュー向け: `${CLAUDE_SKILL_DIR}/patterns-pr-review.md`

Slack向けの場合、共通patternsの一部（ラベル+改行構造、体言止めの羅列、見出し化など）はドキュメント体を前提とした指針なので、Slack patternsの指示が優先される。
PRレビュー向けの場合、詳細説明とGitHubへ貼るレビュー草案を分けて扱い、草案から内部用の優先度ラベルとレビュー過程を除く。

### 3.5 機械的カウント（必須）

目視のチェックは見落としが起きやすい。`${CLAUDE_SKILL_DIR}/check.py` を使って件数を機械的に数えてから、修正方針を判断する。
「保持判断だから件数にカウントしない」と内部で判定してスキップしない。検出はあくまで件数報告までで、保持/除去の判断はユーザーと合意のうえ行う。

```bash
python3 "${CLAUDE_SKILL_DIR}/check.py" <対象ファイル>
# PRレビュー向け
python3 "${CLAUDE_SKILL_DIR}/check.py" <対象ファイル> --pr-review
```

スクリプトはコードブロック・インラインコード・URL を自動的に除外して、以下のパターンの件数をマークダウン表で出力する。

- 箇条書きラベル `- xxx: 説明`
- 散文中ラベル `xxx: 説明`
- 日英間スペース（助詞 → 英単語）
- 日英間スペース（英単語 → 助詞）
- 「」での強調
- （）での補足

`--pr-review`では次も検出する。

- `must-fix` / `should-fix` / `watch` / `P0`〜`P3`
- `review-pr` / `triage` / reviewer名などの内部処理
- `〜ください`を使った強い依頼
- `🙏`直前の不要なスペース
- 太字による強調
- お願いがある草案で、文末の`🙏`が抜けているケース
- 文末以外に`🙏`があるケース

10 件超のパターンが 1 つでもあれば、スクリプトが警告を出す。修正前にリストを見せてユーザーに方針確認する（保持・除去・部分対応のいずれか）。
判断結果は会話に記録し、Step 4 の修正範囲を確定する。
PRレビュー向けの追加パターンは1件でも除去する。内部判定で優先度を保持することと、ユーザー向け文章へ表示することを分ける。

### 4. 修正

意味と共有前提の確認（Step 2）、機械的カウント（Step 3.5）、目視検出（Step 3）の結果を統合し、ユーザーと合意した方針で Edit で修正する。
1 パターンあたり 10 件以下なら確認なしで修正してよい。10 件超は方針合意済みの範囲のみ実行する。
日英間スペースなど大量置換が必要な場合は Python script で一括処理し、コードブロック・インラインコード・URL を保護する。

### 5. Reader Testing

修正後、Subagentに修正結果を読ませ、理解と文体を別々に確認する。どちらにも修正意図や期待する正解は伝えない。

```
Agent(subagent_type="general-purpose"):
  "あなたは<想定読者>です。共有済み前提は<前提>です。
   以下のテキストだけを読み、次を報告してください。
   - 中心的な主張または変更を1文で要約
   - 決定済み、提案中、未確認の区別
   - 根拠を辿れない主張、未定義語、補完が必要な因果関係
   - 読後に必要な次の行動
   - 情報不足の診断が本文へ混ざり、文章の目的を変えていないか
   - 文体が急に変わる箇所、反復、不自然に説明的な箇所"
```

Reader Testingの前に、この文章で必須となる理解を3〜5項目にし、最低1項目をcriticalとして固定する。criticalが`部分的`または`未達`なら完了せず、校正案を修正する。元情報の不足で修正できない場合は、不足を確認事項としてユーザーへ返す。

想定読者や共有済み前提を特定できない場合は、文書種別と共有先から推定する。推定によって修正内容が大きく変わる場合はユーザーへ確認する。構文パターンの網羅はStep 3.5、理解成立の確認はこのReader Testingが担う。

### 6. 完了確認

Step 3.5で報告した全パターンが「ユーザー合意の方針通り処理済み」になっていることを再度 `check.py` で確認する。

```bash
python3 "${CLAUDE_SKILL_DIR}/check.py" <対象ファイル>
# PRレビュー向け
python3 "${CLAUDE_SKILL_DIR}/check.py" <対象ファイル> --pr-review
```

「除去」と合意した項目は件数 0 になっているか確認。残件がある場合は処理漏れなので追加修正する。「保持」とした件は残っていてよい（その旨をユーザーに報告）。

問題があれば追加修正。なければ完了。

## Gotchas

- コードブロック内のコメントは校正対象外。コード中のコメントはそのプロジェクトのスタイルに従う
- CLAUDE.mdやSKILL.mdはClaude向けの指示文なので、ユーザーのトーンに寄せすぎない。ただしAIっぽいパターンは除去する
- 英語のテキストはスコープ外。日本語部分のみ対象
- PRレビュー向けでは、お願いを含む各コメントの文末にスペースなしで`🙏`を付ける。事実だけのコメントには付けない
