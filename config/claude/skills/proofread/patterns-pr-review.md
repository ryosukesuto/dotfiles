# PRレビュー向け文体

PRレビューの説明、インラインコメント草案、Approve / Request changes本文に適用する。

## 表示ルール

- `must-fix`、`should-fix`、`watch`、`P0`〜`P3`は内部判定にだけ使い、ユーザー向けレビュー結果とGitHub投稿文には出さない
- reviewer名、confidence、triage、dedupe、使用skillなど、レビュー内部の処理を書かない
- 各指摘には対象箇所、詳細説明、根拠、修正の方向、レビュー草案を付ける
- 詳細説明では「現在の挙動」「問題になる条件」「起きる影響」を具体的に書く

## レビュー草案

- 1コメントにつき論点と依頼は1つ
- 2〜4文を目安に、事実 → 気になる理由 → お願いの順で書く
- 日本語と英数字の間に半角スペースを入れない。英単語同士のスペースは残す
- `です・ます`だけで揃えず、`〜かなと`、`〜したいです`、`〜気になりました`を混ぜる
- `〜ください`を使った強い依頼を避け、相談・提案として書く
- お願いを含むコメントは、文末にスペースを入れず`🙏`を付ける
- 事実を伝えるだけのコメントに`🙏`を付けない
- `🙏`は1コメントにつき1個まで
- 太字、引用以外の「」、ラベルで始まる前置きは使わない
- 技術的な識別子、ファイル名、設定値はバッククォートで囲む

## 出力例

```markdown
ここ、`workflow_dispatch --ref`だとbranch側のworkflowが使われます。workflow内のguardもbranch版から外せるので、repository secretsに触れる経路が残るのが気になりました。

default branch版だけを使うtriggerへ置き換えるか、secretsをprotected environmentに移したいです🙏
```

## 最終確認

- 内部用の優先度ラベルが残っていない
- 根拠のない断定や、実装者の意図を決めつける表現がない
- 詳細説明とレビュー草案で同じ文章を繰り返していない
- お願いの文末だけに`🙏`が付いている
