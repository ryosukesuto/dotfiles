## 検出パターン

優先度順。上ほど検出されやすく、修正の効果が大きい。

### P1: 構文パターン

| パターン | 修正方針 | 例 |
|---------|---------|-----|
| 「」で強調 | バッククォートか、そのまま削除 | 「publicだから入れられない」→ publicだから入れられない |
| （）で補足 | 「—」「、」に置き換えるか別文に分ける | 設定（git, ssh等）→ 設定 — git, ssh等 |
| この〜が〜を〜する | 事実を並べる。因果は読み手に委ねる | この制約が構造化を強制する → publicだと機密を入れられない |
| 一文目の帰結を二文目で言い換え | 二文目を削って次の事実に繋げる | 機密を入れられない。自然と分離が進む。→ 削除して次の事実へ |
| 同じ概念を別の言葉で繰り返す | 片方を削る | 冪等で、何度実行しても安全 → 冪等 |
| 理由→結論→結論の言い換え、と3段で説明 | 理由+結論の1文にまとめる | checkoutは使わない。並行なので独立ディレクトリが要る。同時に触れる → 並行で進めるため、通常のブランチは封印 |
| 〜を定義し、〜と連携させている | 読点で繋ぐ | 〜を定義し、〜と連携 → 〜を定義、〜と連携 |
| `:` で終わるラベル表現 | 見出しや箇条書きで構造化する。散文中の `:` は「—」か改行に置き換え | 環境によって異なる: → 見出し or 改行して箇条書き |
| 箇条書きの `ラベル: 説明文` パターン | ラベルを削除して説明文だけにする。ラベルなしで意味が通じるなら不要 | - 書かない: 人間が書くのは〜 → - 人間が書くのは〜 |
| 日本語とアルファベットの間にスペース | スペースを入れない | server リポジトリ → serverリポジトリ |
| ガイドライン・手順書で「〜する。」と文末を丁寧にする | 体言止めにする。散文・説明文では「〜する。」のままでよい | 5項目を判定する。 → 5項目を判定 |

### P1.5: 抽象化パターン

| パターン | 修正方針 | 例 |
|---------|---------|-----|
| 説明文で冒頭を書く | 名詞句のキャッチコピーにする。動詞を含まない | Claude Code / Codex CLIの設定と制御をdotfilesで管理する → AI開発特化型dotfiles |
| 汎用的な説明で特徴を隠す | 何が他と違うかを具体的に書く | PRレビューとフィードバック → Claude Code+Codexの協業PRレビュー |
| 数や量で特徴を示そうとする | 数は省略。構造で示す | 30以上配置。文脈に応じて自動で選ぶワークフロー定義 → 配置 |
| 因果を抽象ラベルで説明する | 事実を並べて因果は読み手に委ねる | この制約が構造化を強制する → 削除 |

AIは正しいが誰にでも当てはまる説明を書きがち。読み手が知りたいのは、何が他と違うのか。

### P2: 語彙パターン

| パターン | 修正方針 |
|---------|---------|
| 特徴的な部分 | 削除するか具体的に言い換え |
| 一元管理している | 管理している、まとめている |
| 体系的に | 削除 |
| 自律的に | 自動で |
| 再利用可能な | 削除するか具体的に |
| 設計思想は〜 | 削除して事実だけ書く |
| 行動ルール | ルール（「行動」は不要） |
| 利用しない | 封印、使わない（意志が出る表現を選ぶ） |
| フック（カタカナ） | hook（英語のまま） |

### P3: 構造パターン

| パターン | 修正方針 |
|---------|---------|
| 見出しにラベル説明を付ける | 見出しは名詞だけ。`Hook: AIの行動にガードレール` → `Hook` |
| 見出しが「〜の〜」で冗長 | 名詞1-2語に削る。`設計の考え方` → `設計` |
| 見出しに動詞句を含む | 名詞句にする。`install.shがやること` → `install.sh` |
| 見出しに読めば分かる情報を入れる | 図やコードで分かることは見出しから省く。`publicリポジトリと機密分離` → `機密情報の分離` |
| 箇条書きの各項目が同じ構文 | 文の長さと構文を混ぜる |
| 冒頭に目的や概要を宣言する | 削除。すぐ本題に入る |
| 見出しの内容を本文冒頭で繰り返す | 見出しで伝わっているなら削除して次の事実から始める |
| 図やコードブロックで示した内容を散文でも説明する | 図で伝わっているなら散文を削除。セクションごと不要になることもある |
| 末尾にまとめ・次のステップ | 不要なら削除 |
| 箇条書きの項目数が多い | 3-4項目に絞る。全部列挙しなくていい |
| READMEにセットアップの全手順を書く | 最低限のコマンドだけ。手動設定の詳細はコード内コメントやテンプレートに任せる |

### 判断基準

迷ったら元の文を残す。過剰に直すと別の不自然さが出る。

修正の目的はユーザーのトーンに合わせることであって、文章を短くすること自体が目的ではない。ただし、図・コード・見出しで既に伝わっている内容の繰り返しは積極的に削る。
