## 検出パターン

優先度順。上ほど検出されやすく、修正の効果が大きい。

### P1: 構文パターン

| パターン | 修正方針 | 例 |
|---------|---------|-----|
| 「」で強調 | バッククォートか、そのまま削除 | 「publicだから入れられない」→ publicだから入れられない |
| （）で補足 | 「—」「、」に置き換えるか別文に分ける | 設定（git, ssh等）→ 設定 — git, ssh等 |
| この〜が〜を〜する | 事実を並べる。因果は読み手に委ねる | この制約が構造化を強制する → publicだと機密を入れられない。分離が進む |
| 〜ための〜 | 目的語を先に置いて体言止め | 機密分離の強制力を得るため → 機密を分離する手段 |

### P2: 語彙パターン

| パターン | 修正方針 |
|---------|---------|
| 特徴的な部分 | 削除するか具体的に言い換え |
| 一元管理している | 管理している、まとめている |
| 体系的に | 削除 |
| 自律的に | 自動で |
| 再利用可能な | 削除するか具体的に |
| 設計思想は〜 | 削除して事実だけ書く |
| 〜を定義し、〜と連携させている | 〜を定義。〜と連携 |

### P3: 構造パターン

| パターン | 修正方針 |
|---------|---------|
| 見出しにラベル説明を付ける | 見出しは名詞だけ。`Hook: AIの行動にガードレール` → `Hook` |
| 箇条書きの各項目が同じ構文 | 文の長さと構文を混ぜる |
| 冒頭に目的や概要を宣言する | 削除。すぐ本題に入る |
| 末尾にまとめ・次のステップ | 不要なら削除 |

### 判断基準

迷ったら元の文を残す。過剰に直すと別の不自然さが出る。

修正の目的はユーザーのトーンに合わせることであって、文章を短くすること自体が目的ではない。情報量は減らさず、表現だけ変える。
