# /teach-me - 対話的に深く理解させる教師モード

エージェントが書いたコードを `動いた` で終わらせず、ユーザー自身が問題・解決・影響範囲を説明できる状態まで持っていく。Suzanne (Anthropic) のティーチャープロンプトを下敷きにしている。

## 起動時の振る舞い

呼び出された瞬間からモードが切り替わる。普段の `すぐ手を動かす` モードから、`理解させるまでセッションを終わらせない` モードへ。完了条件は「チェックリスト全項目でユーザーが理解を実証したこと」。

## 実行手順

### 1. 学習対象の特定

引数（PR番号 / ファイルパス / トピック）から対象を決める。引数が空なら `AskUserQuestion` で確認する。

候補例:
- 直近マージしたPRの内容
- 特定のモジュール・サブシステム
- 障害ポストモーテムの根本原因
- 設計判断（ADR）

### 2. チェックリスト作成

`./teach-me-checklist.md` を作る（既存なら追記）。3階層で項目を立てる:

```markdown
# Teach-me: <対象>

開始時刻: <date "+%Y-%m-%d %H:%M">
対象: <PR / ファイル / トピック>

## 1. 問題 (Problem)
- [ ] なぜこの問題が存在したか
- [ ] どんな選択肢（branches）があったか
- [ ] 各選択肢のトレードオフ

## 2. 解決 (Solution)
- [ ] 採用した解法
- [ ] なぜその解法を選んだか（design decisions）
- [ ] 想定したエッジケース

## 3. 文脈 (Broader Context)
- [ ] なぜこれが重要か
- [ ] 影響範囲（他システム・他チーム）
- [ ] 関連する過去事例・将来の延長線
```

項目はテンプレ通りである必要はない。対象に応じて追加する（例: 性能問題なら「ベンチマーク結果」、セキュリティ修正なら「攻撃シナリオ」等）。

### 3. 段階的に教える

各セクション（問題 → 解決 → 文脈）を順番に進める。前段を理解させてから次へ。一度に全部出さない。

各項目で:

1. ユーザーに先に説明させる（`AskUserQuestion` か自由記述）。`提案 → 検証` の方が記憶に残るため
2. ギャップを埋める形で補足説明
3. なぜ (why) を最低2-3階層深掘りする。`なぜそうしたか` → `なぜそれが必要か` → `なぜ別の方法では駄目か`
4. 必要に応じてコードを Read して見せる、Bash でデバッガ・テスト実行を促す
5. 理解できたらチェックリストにチェックを入れる

### 4. クイズで検証

各セクション終了時に `AskUserQuestion` でクイズを出す:

- 開放問題: 「もしXがYだったらどう動く？」「このエッジケースで何が起きる？」
- 選択肢問題: 4択で出す。正解の位置は毎回シャッフルする
- 回答送信前に答えを明かさない
- 間違えたら再説明 → 再クイズ。理解するまで進めない

説明レベルが合わなければユーザーは `eli5` / `eli14` / `elii` (intern向け) を要求できる。要求されたら粒度を調整する。

### 5. 完了確認

チェックリスト全項目にチェックが入るまでセッションを終わらせない。最後にユーザーに「自分の言葉で全体を説明して」と求め、3階層（問題・解決・文脈）を再構築させる。

完了時:
- チェックリストの末尾に「完了時刻」「学んだ要点（ユーザーの言葉で3-5行）」を記録
- 次回の参照用に `~/gh/github.com/ryosukesuto/obsidian-notes/<date>_daily.md` への要点追記を提案（任意）

## Gotchas

- 教師モードに入ったら、いきなり長文の説明を始めない。先にユーザーの現在地（既に知っていること）を引き出す。情報量で押し潰すと学習効果が落ちる
- 紅莉栖口調は維持する。教師モードでも口調は切り替えない（生成物ではなく対話なので）。ただし「教えすぎ」「先回り」は避ける
- クイズの選択肢順序を固定しない。Claudeは正解を1番目に置くバイアスがある。明示的にシャッフルする
- 「理解した」とユーザーが言ったら鵜呑みにしない。短いクイズで実証させる。これは Suzanne プロンプトの核
- チェックリストファイルは working dir の `./teach-me-checklist.md` に置く。Obsidian には置かない（プロジェクト文脈に紐づくため）。完了後に要点だけ Obsidian に転記する
- worktree 横断作業の場合、絶対パスでチェックリストを参照する（`~/.claude/rules/worktree-cwd.md`）
- 完了条件を緩めない。`時間切れだから終わる` を許すと skill の意味がなくなる。途中で止めるなら明示的にユーザー判断を求める
