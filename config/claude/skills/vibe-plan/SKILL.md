# /vibe-plan - 実装計画の作成

design.md（設計書）を基に、実装計画（plans.md）を作成します。

## 前提条件

- design.md がCodexレビューでOKになっていること
- `/vibe-review` でMajor issueがないことを確認済み

## ワークフロー概要

```
design.md確定 → [vibe-plan] → plans.md作成 → Codexレビュー → 実装開始
```

## ファイル管理ルール

- plans.md は design.md と同じ `docs/` ディレクトリに作成
- 命名規則: `docs/YYYY-MM-DD_機能名_plans.md`（design.md と日付・機能名を揃える）

## 実行手順

### ステップ1: design.md の確認

設計書を読み込み、以下を把握：

1. 目的・ゴール
2. 提案するアーキテクチャ
3. 変更対象のコンポーネント
4. 非機能要件

### ステップ2: 変更範囲の特定

コードベースを調査し、具体的な変更ファイルを特定。

### ステップ3: plans.md の作成

以下のテンプレートで実装計画を作成：

```markdown
# タイトル（実装計画）

## 1. 目的と概要
- 対応する design.md: [リンクまたはパス]
- 一言サマリ:
- 実装のゴール:

## 2. 変更範囲
### 2.1 システム / コンポーネント
### 2.2 ファイル / ディレクトリ

## 3. Sprint Contract（完了条件）

Sprint単位で「何をもって完了とするか」をテスト可能な形で定義する。
Evaluator（Codex）との事前合意として機能し、実装後の検証基準になる。

### Sprint 1: [機能名]
完了条件:
- [ ] [具体的な動作]: [検証方法]
- [ ] [具体的な動作]: [検証方法]

### Sprint 2: [機能名]
完了条件:
- [ ] [具体的な動作]: [検証方法]

## 4. タスク分解
### 4.1 実装タスク
- [ ] タスク1
- [ ] タスク2

### 4.2 非機能対応タスク
- [ ] ログ出力追加
- [ ] メトリクス追加
- [ ] テスト追加

## 5. PR 方針
- PR の分割単位
- それぞれのPRに含める内容
- レビュー観点のメモ

## 6. マイグレーション計画
### 6.1 手順
### 6.2 ロールバック手順
### 6.3 データに関する注意点

## 7. 検証・テスト計画

## 8. リスクと対策（実装フェーズ視点）

## 9. スケジュールと工数目安
```

### ステップ4: Sprint Contract作成のポイント

Sprint Contractは「実装前にEvaluator（Codex）と完了条件を合意する」仕組み。自己評価バイアスを排除し、実装後の検証を客観的にする。

1. 完了条件は検証可能な動作で記述する（「〜ができる」「〜が返る」「〜が表示される」）
2. 曖昧な基準（「適切に動作する」）は禁止。具体的な入出力やUIの状態で定義する
3. Sprint単位は1PR〜2PRに収まるサイズ。依存関係順に並べる
4. ロールバック可能な単位を意識する

## 出力物

1. `plans.md` - 実装計画書

## 次のステップ

plans.md 作成後、`/vibe-review` でCodexにレビューを依頼してください。

OKになったら、以下の手順で実装を開始：

1. `docs/coding-guidelines.md` を読み込む（コーディング規約の確認）
2. plans.md のタスクに従って実装
3. 自己レビュー（lint/test/設計整合性）
4. PR作成 → `/review-pr`
