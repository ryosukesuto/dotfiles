# /explainer-doc - 解説ドキュメント生成

技術設計・議事録・ADR・設計書を、ドメイン外の人が読んで理解できる解説ドキュメントに変換する。汎用ドキュメントとして書き、特定の対象（新卒向け等）は明示しない。

## 実行手順

### 1. ソース特定

引数または会話文脈から元情報を集める:

- 議事録: `Glob "*<topic>*"` でObsidian内検索
- ADR / 設計書: GitHub URLを `gh api repos/<org>/<repo>/contents/<path>?ref=<sha>` で取得
- 会話に直接含まれているトピックなら追加取得不要

複数ソースで記述が食い違う場合、ADRを正本にする。議事録は音声認識ガーブルが混じる可能性があるため。

### 2. 読み手と理解の更新

想定読者が持つ汎用知識・組織知識と、読後に理解または判断できるようにすることを1文で決める。本文に対象者ラベルを機械的に書く必要はないが、説明量はこの読者モデルから決める。

### 3. 構成

以下を基本構成とし、内容があるセクションだけを使う。空欄や一般論で固定構成を埋めない。

1. このドキュメントの位置づけ — 1段落で要約 + 正本（ADR/議事録/設計書）へのリンク
2. 前提知識 — 想定読者に共有されていない用語だけを記載
3. いま起きている問題 — なぜこの設計が必要か（Mermaidで現状フロー）
4. どう解決するか — 採用案 + トレードオフ（Mermaidで新フロー）
5. 現在のステータス — 検証段階・残課題

必要な場合だけ追加する:

- なぜ他の方法を採らなかったか — 比較が判断の理解に必要な場合
- 関連リソース — 読み手が参照できる正本や関連資料がある場合

サンプル: `/Users/s32943/gh/github.com/ryosukesuto/obsidian-notes/2026-05-02_2リージョン順次デプロイ設計.md`

### 4. 前提知識テーブル

用語が複数あり、比較可能な形にすると理解しやすい場合だけテーブルを使う。汎用用語と組織固有用語は混ぜない。想定読者に共有済みの用語は説明しない。

汎用テーブルの例: Kubernetes / ArgoCD / Helm chart / GitOps 等
組織固有テーブルの例: WinTicket / Bumper / stg / prd / tokyo / osaka 等

ルール:

- 内製ツール（Bumper等）は OSS 相当を併記する。例: 「Bumper は内製のリリース PR 自動作成ツール。`release-please` / `semantic-release` 相当」
- 環境名（stg/prd/dev等）は何のための環境かを書く
- リージョン名は何の上の何かを書く（GCP / AWS のどのリージョン等）

### 5. Mermaid図

`graph TD` で書く。CLAUDE.md `obsidian-mermaid` ルール準拠:

- `flowchart TD` ではなく `graph TD`
- ノードテキストに `1. ` のような番号付きリストを書かない（Markdownリストとして解釈されエラーになる）
- `subgraph` には ID を付ける（`subgraph SG1[ラベル]`）

関係、順序、状態遷移が散文より明確になる場合だけ使う。現状と採用案の差分が重要なら2図、片方だけで理解できるなら1図、図が不要なら入れない。

### 6. ファイル作成

ファイル名: `YYYY-MM-DD_<topic>.md`

- 配置: Obsidian Vault ルート直下（`~/gh/github.com/ryosukesuto/obsidian-notes/`）
- サブフォルダ作成禁止（CLAUDE.md フラット構造ルール）
- フロントマター: `tags: [types/tech]`、`title`、`date`
- 日付は `date "+%Y-%m-%d"` で取得
- topic は短い日本語（例: `2リージョン順次デプロイ設計`）。「新卒向け」等の対象指定は含めない

### 7. インバウンドリンクの追加

孤立ノートにしないため、既存オンボードドキュメントから新ドキュメントへリンクを貼る。

```bash
Glob "*オンボーディング*"
```

該当ファイルが見つかれば、末尾の「関連リンク」セクション直前に「次に読む」セクションを追加:

```markdown
## 次に読む

- [[YYYY-MM-DD_<topic>]] — 1行説明
```

### 8. 校正

`/proofread` で、読み手に必要な理解が成立するかを確認してから文体を整える。

## NG パターン

本文・見出し・タイトルに入れない:

- 「新卒向け」「初心者向け」「初学者向け」のような対象指定の語
- 「学習ポイント」「まとめ」のような締めセクション（AI っぽい）
- 見出しに括弧補足（例: `採用案 (オプション3) の構成` → `採用案の構成`）
- 「〇〇とは」見出し直下で本文冒頭に同じ説明を繰り返す

## Gotchas

- ADR と議事録で記述が食い違うときは ADR を正本にする。議事録は音声認識ガーブルの可能性あり
- Mermaidの矢印ラベルに改行・複雑記号を入れるとObsidianでレンダリング失敗する。シンプルに保つ
- 「現在のステータス」セクションは会話時点の情報。後から読む人が判断できるよう日付を必ず明記する
- 既存オンボードドキュメントを読まずに「次に読む」を追加すると、既存のリンク構造を壊す可能性がある。追加位置を決める前に必ずReadする
- 内製ツールにOSS相当が見当たらない場合、無理に併記しない。代わりに「役割: 〇〇する内製ツール」のように責務を1行で書く
