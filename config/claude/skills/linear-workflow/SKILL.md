# Linearタスク管理ワークフロー

チームのタスク管理ガイドライン。
Linearを「タスク管理ツール」から「意思決定ログ」に変える。

## 階層とライフサイクル

### Initiative / Project の違い

| 階層 | 役割 | 期間 | 完了条件 |
|------|------|------|----------|
| `Initiative` | 戦略単位・中長期の方向性。複数Projectを束ねる「なぜやるか」 | 数四半期〜1年以上 | 方向性のみ、成果物は示さない |
| `Project` | 実装単位・成果物が明確 | 数週〜数か月 | DoD必須、Issueを内包 |

### Issue ステータスの使い分け

```
Triage → Backlog → Todo → In Progress → Done
         （やるかも） （着手待ち）
```

| State | 意味 | 使いどころ |
|-------|------|-----------|
| `Triage` | 仕分け待ち | 外部起票（Slack連携・他チーム依頼）の受信箱。内部で直接起票する場合はスキップ |
| `Backlog` | やる可能性はあるが着手未定 | アイデア段階、将来やるかも。サイクル未計画 |
| `Todo` | やると決定、着手待ち | 直近サイクルで実行予定。スコープ・優先度明確 |
| `In Progress` | 着手中 | - |
| `Done` | 完了 | - |

自分で起票するIssueは`Todo`から始める（`Triage`は経由しない）。

## タスク・プロジェクトの作成

### プロジェクト作成基準
- 複数サイクルにまたがる → Linearプロジェクトとして作成
- プロジェクトには原則ゴールとマイルストーンを設定
- 工数が読みづらい場合でも、期限は一旦決めでOK

### プロジェクト必須項目

通常Projectは以下4点を必ず設定する。金曜の定例でInitiativeを俯瞰する運用に乗せるため、未設定だと俯瞰時に状態が読めない。

#### 1. description先頭にゴールセクション

```markdown
## ゴール
- <達成条件1（状態で書く、完了形）>
- <達成条件2>
- <達成条件3>

## スコープ外
- <やらないこと（1行）>
```

ゴールは3〜5項目、状態で書く（例: `新構成が本番反映済み`、`旧構成が削除済み`）。
スコープ外がないとスコープが膨らんでCloseできない。

#### 2. Milestone

検証フェーズ・段階リリース単位で3〜5個に分ける。1つあたり2〜4週間が目安。

各Milestoneに以下を記載:
- タイトル: 何が起きたら完了かを短く（例: `リハーサル試験①`）
- `目標:` で1行、達成条件
- 日程: 仮でOK。外部イベント等の制約があれば併記

細かく切りすぎると棚卸しコストが上がる。Initiative俯瞰で進捗/停滞が判定できる粒度に留める。

#### 3. Dates（PJ全体の終了目安）

開始日と完了予定日を設定。完了予定日のデフォルトはWinTicket半期末（下期=`YYYY-08-31`、上期=`YYYY-02-末日`）。半期1か月前倒しルールに合わせる。

仮置きでOK。Milestoneの進捗で前倒し/後ろ倒しを反映する。

#### 4. Lead

ProjectのLeadは、そのProjectの完了判断責任者とする。

### 定常運用Projectの扱い

Bug Bounty対応・Wizアラート対応など、ゴールが定まらない定常PJはMilestone・Datesを設定しない。代わりにdescriptionに以下を明記する。

```markdown
## 種別

定常運用PJ。Milestone・終了目安の設定対象外。

## 運用方針

週1でコメントを残す: 今週消費したpt、押し出された作業
```

### イシュー作成基準
- 細々とした依頼 → 単独のイシューとして作成
- Bug Bounty、Wizアラートなど頻出系 → 定常運用Project（上記）にまとめる

### Issue 起票のタイミングと粒度

GitHub PR との自動連携を活かすため、Issue は **着手前に切る**。

| 作業タイプ | Issue 設計 | 自動連携 |
|---|---|---|
| 単発の機能/修正 | 1 Issue = 1 PR | PR description に `Closes PF-XXXX` で自動 Done |
| 複数 PR またぐ施策 | 親 Issue + 子 Issue | 子は `Closes`、親は子が完了したら手動 Done |
| 調査/Spike | 1 Issue（PR不要も可） | 完了時に手動 Done |
| 長期エピック | 親 Issue は In Progress 維持、子を順次切る | 同上 |

着手後に切ると、ブランチ名・PR タイトルを書き直す必要があり手動紐づけ作業が発生する。例外は5分で終わる typo 修正程度。

複数 PR をぶら下げた重い親 Issue（例: 10件以上の PR が紐づく）は、後から子 Issue に分解しなおす必要が出る。最初から子 Issue を切る方が効率がよい。

詳細: `~/.claude/rules/linear-branch-naming.md`

### 通常issueのdescriptionテンプレート

会議系issue（後述）以外の、実装・調査タスクのdescriptionは以下の構成に固定する。セクションを増やしすぎない。

```markdown
## Why

<背景・なぜ今やるか。2-4文>

## スコープ

* <やること>
* <やること>

## スコープ外

* <やらないこと>

## 完了条件

- [ ] <検証項目>
- [ ] <検証項目>

## 依存

<ブロッカーや前提Issueがあれば1-2文。`blockedBy`/`blocks`のリレーションと併用し、リレーションだけでは伝わらない理由を書く>

## 参考

* <関連Issue・PRへのリンク>
```

`スコープ外`・`依存`は無ければ省略してよい。`未確認事項`のような調査中セクションは、会話で解消されたら都度消す（解消済みの問いを残したまま放置しない）。

アンチパターン: セクション見出しに日付を埋め込む（`## スコープ範囲（確認済み、2026-07-19）`等）。Linearは編集履歴とコメントで経緯を追えるため、本文に「いつ確認したか」を書く必要はない。確定した内容は日付を付けず、確定事実としてそのまま`Why`/`スコープ`に書く。同様に「影響範囲（現状把握）」のような調査ログ然としたセクションを別立てにせず、調査で分かった内容はそのまま`スコープ`の各項目に織り込む。

### Claude Codeでのissue作成時の必須項目

Linearでissueを作成する際は、以下の項目を必ず設定する：

| 項目 | 設定値 | 備考 |
|------|--------|------|
| Team | `Platform` | Linearでのチーム名（PFチーム） |
| Assignee | `me`（自分） | 指定がなければ自分をアサイン |
| Estimate | ポイント見積もり | 下記の基準に従う |
| Cycle | 指定なし | 計画時に設定、割り込みは発生時点のcurrent Cycleを指定 |
| State | `Todo` | 初期状態（Triageではなく） |
| Project | 関連プロジェクト | 該当するものがあれば設定 |

Descriptionに「Why 2行」を書く（特にClaude Code生成時）：
- なぜ今やるか
- 何を防ぐか

description・コメントはPR本文と同じ「組織共有ドキュメント」として扱う。個人skill名・Codex等セカンドオピニオンツール名・個人dotfilesパス等、チームメンバーが参照できない語彙は書かない。詳細: `~/.claude/rules/pr-writing.md`

### 起票前の重複確認

同じ内容のIssueが既に存在しないか、作成前に`list_issues`等で検索する。重複Issueは管理コストを増やすだけでなく、どちらが正か分からなくなり手戻りの原因になる。

### スコープはコードを見てから書く

「〜のはず」で推測して書かず、対象のコード・設定ファイル・関連リポジトリを実際に読んでからスコープを確定する。推測でスコープを書くと、着手時に前提が崩れて手戻りになる（例: 参照先ファイルが実は存在しない、依存関係が逆だった等）。

### 完了条件だけをTODOチェックリスト形式で書く

`## 完了条件`（Definition of Done）はプレーンな箇条書き(`*`/`-`)ではなく`- [ ]`のチェックボックス形式で書く。実装が終わって検証が済んだ項目にだけチェックを入れる、Issueをcloseしてよいかのゲートとして機能させる。

`## スコープ`はプレーン箇条書きのままにする。スコープは「これから何をやるか」の計画であり、実装しながら変わりうるもの（着手後に前提が崩れてスコープ自体を書き直すことは珍しくない）。ここをチェックボックス化すると、「実装作業を進めた」ことと「完了条件を満たすことを検証した」ことが混同され、完了条件を一つも検証していない段階でも見た目の進捗率だけが上がってしまう。

`## Why`・`## 未確認事項`・`## 参考`も同様にプレーン箇条書きのままでよい（チェックボックス化するとノイズになる）。

判断基準: 「Issueをcloseしてよいかの検証ゲート」だけがチェックボックス、それ以外の説明・計画・参照情報はプレーン箇条書き。

2026-07-19、CyberAgentCard/processing-infrastructure PFIF-122〜124作成時に一度`スコープ`・`完了条件`の両方をチェックボックス化したが、両方チェックボックスだと役割が重複し進捗の見た目が実態とズレることに気づき、`完了条件`のみに絞った。

### 完了条件の各行は検証項目のみ、根拠は書かない

完了条件のチェックボックス行には「何を検証すればよいか」だけを書く。「済んだ理由」「調査で分かった数値」「参照元URL」等の根拠説明をチェックボックス行に埋め込まない。根拠は`参考`（レポート・PR・ドキュメントへのリンク）や`Why`に置き、完了条件からは辿れるようにする。

判断基準: チェックボックス行を読んで「検証済みかどうか」が一目で分かるか。行が長くなってきたら根拠を`参考`に逃がす合図。

2026-07-19、WinTicket/PF-2620の完了条件更新で「一覧化」項目に棚卸しレポートの詳細（761行・109データセット・分類内訳等）を1行に詰め込み、須藤様から「完了条件の文章が長くない？」と指摘された。根拠は`参考`のレポートリンク1行に集約し、完了条件は`- [x] 個人ユーザー直接付与の一覧化（dataset・付与者）`まで短縮して解消した。

### 関連Issueの依存関係を明示する

複数Issueが順序依存する場合（例: 改称Issueを先に終わらせてから移行Issueに着手する等）、`blockedBy`/`blocks`で関係を設定し、description内にも「## ブロッカー」等で人が読んでも分かるように明記する。

Estimateの目安：
- 対応完了済み（記録目的）: 1 pt
- 軽微な作業: 1-2 pt
- 通常の作業: 2-4 pt
- 大きめの作業: 4-8 pt

### 割り込みIssue

割り込み対応は別Issueにする。その際、影響先を必ず明記：

```
Impact: [止めたProject名] の Issue #xxx を後ろ倒し
```

影響を言語化しないと、割り込みが「見えない負債」になる。

## 各サイクルでの動き

### 週次フロー
1. 各自: 今週のタスクを整理、ポイントづけ（他のタイミングで行えるなら任意参加）
2. 金曜日 60分の定例:
   - 前半30分: Project軸の振り返り（下記参照）
   - 後半30分: 来週やること共有、質問

### Cycle Review（前半30分）

見る対象はProjectだけ。Issueの消化数は原則見ない。

確認項目：
- 今週Closeまたは前進したProject
- 止まったProject
- 止まった理由（割り込み / 見積ミス / 依存）
- 今週、意図的にやらなかったこと（1つ）

最後の「やらない判断」が意思決定ログの核。
例: X ProjectのIssue #123は、Yを優先するため次Cycleへ送った

### 定例参加メンバー
- PFチーム + その週にPFタスクで稼働する兼務メンバー

### タスク割り振り
- プロジェクト以外の細々とした依頼 → PFころ + 挙手で割り振り
- チケット切ってアサインしていく

### 目指す状態
今週やることはLinearのactive cycleを見ればわかる状態

## ポイント見積もり

| 基準 | 目安 |
|---|---|
| 1週間の作業量 | 8 pt |
| 1営業日の作業量 | 2 pt |

- 兼務などで稼働が少ない週は上限をよしなに調整
- 工数はブレる想定なので、取り組む前の体感で決めてOK

## Linear MCP API の注意点

サイクルやチームの指定には、名前やキーワードではなくIDを使う。

### サイクル操作の手順
1. `list_cycles(teamId, type="current")` でサイクルIDを取得
2. `list_issues` でサイクル指定する場合は、取得したIDを使う（`current` 等の文字列は効かない）
3. issueのサイクル移動も同様にIDを直接指定（`next` 等の文字列は効かない）

```
# 正しい手順
cycles = list_cycles(teamId="<TEAM_ID>", type="current")
cycle_id = cycles[0].id
list_issues(assignee="me", cycle=cycle_id)

# 間違い（効かない）
list_issues(assignee="me", cycle="current")
update_issue(id="PF-xxx", cycle="next")
```

## 会議系issueのテンプレート

キックオフ・レビュー会議などを Linear に登録するときの description テンプレート。

### テンプレート

```markdown
## 目的

<会議の目的を1行>

## 日時

YYYY-MM-DD（曜）HH:MM-HH:MM（NN分）

## 資料

[<タイトル>](<Kibela URL>)

## 参加者

- @username1
- @username2
- @suto_ryosuke_xa

## アジェンダ

1. <議題1>
2. <議題2>

## 次のアクション

- [ ] <アクション1>
- [ ] <アクション2>
```

### 注意点

- Kibelaリンクは必ずMarkdownリンク形式 `[タイトル](URL)` で書く。URL単体（`Kibela: https://...`）だとLinearが attachments に自動吸収して description 本文から消える
- attachments欄に自動で追加される分はそのまま利用してOK。description内のMarkdownリンクと併存できる
- 参加者は `meeting-minutes` スキルの `convert-participants.py` で NASCAテキストから変換できる

## やらないこと

運用が目的化するのを防ぐため、以下は意図的に入れない：

- 細かいステータスを増やす
- pt実績を厳密に追う
- KPIっぽい数値評価
