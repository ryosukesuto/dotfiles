## 監査チェックリスト

各カテゴリの具体的な確認項目。存在 / 品質 / 自動化の3段階で評価する。

### スコア 0 vs 1 の共通判別基準

全カテゴリに適用する下限の判別ルール:

- 0（未構成）: 該当カテゴリに関連するファイル・設定が1つも存在しない
- 1（最小限）: 関連ファイル・設定は1件以上存在するが、品質が最低限で自動化されていない / 形骸化している

判断に迷う場合は、「そのカテゴリで最低限期待される成果物が1件でも見つかるか」で切り分ける。見つかれば 1、見つからなければ 0。

### A. コンテキスト設計

エージェントが正しい判断をするための情報構造。CLAUDE.mdはエージェントにとっての地図であり、地図の品質が行動の質を決める。

| 確認項目 | 観点 |
|---------|------|
| CLAUDE.mdが存在する | ビルド・テスト・デプロイコマンドが記載されているか |
| サイズが適切（50行以下推奨） | 肥大化していないか。150行を超えると primacy bias で後半の指示が無視されやすい（IFScale 研究） |
| 段階的開示 | 詳細は `docs/` や `.claude/rules/` に分離されているか |
| アーキテクチャが記述されている | ディレクトリ構造・レイヤー・依存関係が明記されているか |
| 非自明なパターンが記録されている | Gotchas、暗黙の前提、過去の失敗が文書化されているか |
| ルールファイルが活用されている | `.claude/rules/` に判断基準が分離されているか |
| ドキュメントが実行可能 | コマンドがコピペで動くか。抽象的な説明だけになっていないか |

得点の目安:
- 5: 段階的開示が機能し、ルール・Skillが整備されている
- 3: CLAUDE.mdはあるが単一ファイルに詰め込まれている
- 1: CLAUDE.mdはあるがビルドコマンド程度
- 0: CLAUDE.mdがない

### B. フィードバックループ

エージェントの出力を速く・自動で検証する仕組み。速いフィードバックほど価値が高い。

| 確認項目 | 観点 |
|---------|------|
| PostToolUse Hook | ファイル保存後に自動フォーマット・リンターが走るか |
| PreCommit Hook | コミット前に型チェック・リンター・テストが走るか |
| リンター設定 | プロジェクトに適したリンターが設定されているか |
| 型チェック | TypeScript strict / mypy / Go vet 等が有効か |
| テスト | ユニットテスト・統合テストが存在し実行可能か |
| CI/CD | PRマージ前にテスト・リンター・型チェックが走るか |
| E2Eテスト | Playwright / bats-core / Hurl 等が設定されているか |
| PR自動レビュー（Greptile等） | `.greptile.yaml` が存在し、custom standards でプロジェクト固有ルールが設定されているか |
| 成功時のサイレント性 | Hook・CI が成功時に不要な出力を出していないか。pass ログの氾濫で context を溢れさせていないか（Success should be silent; only failure should make noise） |

フィードバックの速度階層（速い順）:
1. PostToolUse Hook（ミリ秒）
2. フォーマッター / リンター（秒）
3. 型チェック（秒）
4. ユニットテスト（秒-分）
5. PreCommit Hook（秒-分）
6. CI（分）
7. E2Eテスト（分）
8. PR自動レビュー（Greptile等）（分-時間）
9. 人間レビュー（時間）

得点の目安:
- 5: PostToolUse Hookで即時フィードバック + CI + E2E
- 3: リンター + テスト + CIはあるがHookは未設定
- 1: CIのみ
- 0: 自動検証なし

### C. アーキテクチャ制約

構造で品質を強制する仕組み。ルールは「提案」ではなく「強制」する設計が効果的。

| 確認項目 | 観点 |
|---------|------|
| 型システムの厳格さ | strict mode / noImplicitAny 等が有効か |
| 依存方向の制約 | レイヤー間の依存が構造テスト等で強制されているか |
| インポート制約 | 循環依存の検出、パッケージ境界の強制 |
| ビルド制約 | unused imports / dead code の検出が自動化されているか |
| ADR（Architecture Decision Records） | 設計決定の履歴が保全されているか |

得点の目安:
- 5: 構造テスト + 依存制約 + ADR が運用されている
- 3: strict mode + 基本的なリンタールール
- 1: 型システムはあるが緩い設定
- 0: 型チェックなし / 構造的制約なし

### D. エントロピー管理

リポジトリの劣化を検出・修復する仕組み。エージェントは古い情報と新しい情報を区別できない。

| 確認項目 | 観点 |
|---------|------|
| ドキュメントの鮮度 | CLAUDE.mdの内容がコードの現状と一致しているか |
| デッドルールの検出 | 古い・矛盾するルールが残っていないか |
| 未使用コードの管理 | dead code、unused dependencies が放置されていないか |
| リポジトリ衛生 | `.gitignore` が適切か。不要ファイルがコミットされていないか |
| 定期クリーンアップ | 型エラー・Lint違反の定期スキャンが仕組み化されているか |

得点の目安:
- 5: 自動スキャン + 定期クリーンアップPRが運用されている
- 3: 手動でたまに整理している
- 1: 一応整備されているが陳腐化が進んでいる
- 0: メンテナンスの仕組みがない

### E. 計画と実行の分離

エージェントの行動を段階的に制御する仕組み。計画なしの実行はスコープクリープを招く。

| 確認項目 | 観点 |
|---------|------|
| Plan Mode活用 | 非自明なタスクで計画フェーズを経ているか |
| Skill整備 | 繰り返しワークフローがSkill化されているか |
| Subagent活用 | 独立タスクの並行実行・コンテキスト分離ができているか |
| レビューフロー | 実装前レビュー / コミット前レビュー / PR前レビューの構造 |
| タスク分解 | 大きなタスクが検証可能な小単位に分解されているか |

得点の目安:
- 5: Skill + Subagent + 多段レビューフローが確立
- 3: Plan Modeは使うがSkill化は限定的
- 1: 都度プロンプトで指示
- 0: 計画なしに実装を開始する

### F. ガードレール

エージェントの暴走を防ぐ安全装置。信頼しつつ検証する（Trust but verify）の思想。

| 確認項目 | 観点 |
|---------|------|
| 権限制御 | `allowed-tools` で必要最小限のツールに制限しているか |
| mainブランチ保護 | 直接コミット防止（pre-commitフック、ブランチ保護ルール） |
| ループ検出 | 同じエラーの繰り返しを検出・中断する仕組みがあるか |
| 破壊的操作の防止 | `rm -rf`、`git push --force` 等へのガードがあるか |
| セキュリティ制約 | 認証情報のハードコード防止、`.env` のgitignore |
| 評価者の独立性 | コードを書くエージェントとレビューするエージェントが別 session / 別 context で動いているか。自己評価させると中程度の品質でも自信満々に褒める癖があり、誤 Approve の温床になる（Generator-Evaluator 分離） |

得点の目安:
- 5: Hookでループ検出 + ブランチ保護 + 権限最小化
- 3: ブランチ保護 + 基本的なセキュリティ制約
- 1: `.gitignore` のみ
- 0: 保護なし
