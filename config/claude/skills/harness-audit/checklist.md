## 監査チェックリスト

各カテゴリの確認項目。採点には共通ラダーを使い、監査者が変わっても同じ点が付く状態を目指す。

## 共通0-5ラダー

全カテゴリに同じラダーを適用する。各カテゴリの個別目安は設けず、以下の共通定義で統一する。

| スコア | 状態 | 判別基準（どのカテゴリでも同じ） |
|-------|------|-----------------------------|
| 0 | 未構成 | 該当カテゴリに関連する成果物が1件も存在しない |
| 1 | 形骸化 | 成果物は存在するが実効性がない（陳腐化・緩すぎる設定・運用されていない） |
| 2 | 手動運用 | 人が都度実行・保守する形で運用されている。自動化されていない |
| 3 | ローカル自動化 | Hook・リンター・テスト等でローカルに自動検証される |
| 4 | 強制ゲート | CI・ブランチ保護・PRテンプレート・プリコミット等で違反をブロックする仕組みがある |
| 5 | 継続改善と計測 | 効果を計測し、閾値・ルールを更新するサイクルが回っている |

判断に迷う場合は「その成果物がどの段階まで強制力を持っているか」で切り分ける。

### Evidence（根拠）の記録

- 採点時は、根拠となるファイルパス・設定名・コマンド出力を1-2件メモする
- レポートの「状態」列には最有力の Evidence を1件含める（例: `3: PostToolUse hook設定あり（.claude/settings.json:L12）`）
- スコア0 の場合は空欄にせず、探索したパスを明記する形式を使う
  - 形式例: `searched: CLAUDE.md, AGENTS.md, .cursor/rules/; none found`
  - これにより「探索漏れ」と「本当に未構成」を区別できる

### Confidence（確信度）の表記

- `High`: 複数の明確な証拠があり、誤判定の余地がない
- `Medium`: 証拠は1件あるが、運用実態までは確認できていない
- `Low`: 間接的な証拠のみ、または時間不足で探索が浅い

## 単一カテゴリの原則

1つのチェック項目は1つの主カテゴリに割り当てる。横断的に見える項目も主カテゴリを明示する。

| 横断的な項目 | 主カテゴリ | 理由 |
|------------|----------|------|
| 型チェック（strict/mypy/vet） | C. アーキテクチャ制約 | 静的に構造を強制する手段だから |
| 型チェックの実行自動化 | B. フィードバックループ | 検証タイミングの問題だから |
| 評価者の独立性（別session） | F. ガードレール | 誤Approveを防ぐ安全装置だから |
| ループ検出 | F. ガードレール | 暴走防止の安全装置だから |
| ドキュメント鮮度スキャン | D. エントロピー管理 | 劣化検出の仕組みだから |
| PR 2-Approve（AI+人間） | F. ガードレール | 誤マージを防ぐ安全装置だから（副次効果として B も改善） |
| AI/人間経路の非対称設計 | F. ガードレール | `--no-verify` 等での逃げ道を塞ぐ設計判断だから（副次効果として B のフィードバック速度も改善） |

### A. コンテキスト設計

エージェントが正しい判断をするための情報構造。CLAUDE.md・AGENTS.md 等のエージェント向けドキュメント（以下 agent-docs と呼ぶ）の品質が行動の質を決める。

| 確認項目 | 観点 |
|---------|------|
| agent-docs が存在する | ビルド・テスト・デプロイコマンドが記載されているか |
| サイズが適切（50-150行推奨） | 肥大化していないか。150行超で primacy bias により後半の指示が無視されやすい（IFScale 研究） |
| 段階的開示 | 詳細がルール集・docs/・ADR 等に分離されているか |
| アーキテクチャが記述されている | ディレクトリ構造・レイヤー・依存方向が明記されているか |
| 非自明なパターンが記録されている | Gotchas、暗黙の前提、過去の失敗が文書化されているか |
| ルール・スキルが活用されている | 判断基準が分離されたルール集・スキル定義に整理されているか |
| ドキュメントが実行可能 | コマンドがコピペで動くか。抽象的な説明だけになっていないか |

共通ラダーの適用例:
- `5` 継続改善と計測: agent-docs の参照頻度・命中率を計測し、定期的にリファクタしている
- `4` 強制ゲート: ドキュメント鮮度チェックが CI で強制され、陳腐化した agent-docs はマージ不可
- `3` ローカル自動化: agent-docs の lint（リンク切れ・サイズ上限）が PostToolUse / pre-commit で走る
- `2` 手動運用: agent-docs は段階的開示で整理され、人が定期的に更新している
- `1` 形骸化: agent-docs はあるがビルドコマンド程度、または陳腐化して実態と乖離している
- `0` 未構成: agent-docs（CLAUDE.md / AGENTS.md / .cursor/rules 等）が1件も存在しない

### B. フィードバックループ

エージェントの出力を速く・自動で検証する仕組み。速いフィードバックほど価値が高い。

| 確認項目 | 観点 |
|---------|------|
| PostToolUse Hook（または同等の即時検証） | ファイル保存後に自動フォーマット・リンターが走るか |
| PostToolUse Hook の対象範囲 | AI の Edit/Write 直後に差分対象ファイルだけに走るか。全件走査でなく ms オーダーで終わるか |
| PreCommit Hook | コミット前に型チェック・リンター・テストが走るか |
| PreCommit Hook の所要時間 | 5 秒以下に収まっているか。重い check（フルテスト・型チェック）は CI に逃がしているか。重すぎると `--no-verify` で逃げる動機になる |
| リンター設定 | プロジェクトに適したリンターが設定されているか |
| 型チェックの実行自動化 | 型チェックが Hook/CI で自動実行されるか（型システム自体の厳格さは C） |
| テスト | ユニットテスト・統合テストが存在し実行可能か |
| CI/CD | PRマージ前にテスト・リンター・型チェックが走るか |
| E2Eテスト | Playwright / bats-core / Hurl 等が設定されているか |
| PR自動レビュー | Greptile / Claude Code Action 等の自動レビューが custom standards（SKILL.md / config）で運用されているか |
| 成功時のサイレント性 | Hook・CI が成功時に不要な出力を出していないか（Success should be silent) |

フィードバックの速度階層（速い順）:
1. PostToolUse Hook（ミリ秒）
2. フォーマッター / リンター（秒）
3. 型チェック（秒）
4. ユニットテスト（秒-分）
5. PreCommit Hook（秒-分）
6. CI（分）
7. E2Eテスト（分）
8. PR自動レビュー（分-時間）
9. 人間レビュー（時間）

共通ラダーの適用例:
- `5` 継続改善と計測: CI 時間・失敗率・flaky率をダッシュボード化し、速度階層の改善を継続
- `4` 強制ゲート: CI でテスト・リンター・型チェックの全失敗がマージをブロック。AI 経路（PostToolUse）と人間経路（pre-commit / CI）が意図的に非対称設計されており、AI には fast feedback、人間には minimal commit ガードを提供している
- `3` ローカル自動化: PostToolUse / pre-commit でリンター・テストが自動実行される
- `2` 手動運用: リンター・テストは整備済みだが、実行は人が都度行う
- `1` 形骸化: CI 設定はあるが実質動いていない／成功ログが溢れて機能不全。pre-commit が重すぎて `--no-verify` が常用されている
- `0` 未構成: 自動検証の仕組みが1件もない

#### AI/人間経路の非対称設計（`4` 以上で重視する）

「AI には厳しく、人間には緩く」が成立しているかを見る軸。同じ git hook で AI と人間が同じ厳しさを浴びる構成は、人間が `--no-verify` で逃げて AI も検証されない経路に流れる失敗パターンになる。

評価ポイント:

- AI 経路の fast feedback: PostToolUse Hook で AI の Edit/Write 直後に差分対象だけ走る（ms オーダー）
- 共通経路の minimal ガード: git pre-commit は壊れた commit を防ぐ最低限のみ。重い check は CI に逃がす
- CI 経路のフル check: フルテスト・型チェック・統合テストは CI で gate
- 経路分離の意図性: 設計者が AI/人間の非対称性を意識しているか（ドキュメントや commit message から読み取る）

### C. アーキテクチャ制約

構造で品質を強制する仕組み。ルールは「提案」ではなく「強制」する設計が効果的。

| 確認項目 | 観点 |
|---------|------|
| 型システムの厳格さ | strict mode / noImplicitAny 等が有効か（実行タイミングは B） |
| 依存方向の制約 | レイヤー間の依存が構造テスト等で強制されているか |
| インポート制約 | 循環依存の検出、パッケージ境界の強制 |
| ビルド制約 | unused imports / dead code の検出が自動化されているか |
| ADR（Architecture Decision Records） | 設計決定の履歴が保全されているか |

共通ラダーの適用例:
- `5` 継続改善と計測: ADR の運用フローが確立し、制約違反数の推移を計測している
- `4` 強制ゲート: CI で構造テスト・依存制約・unused コード検出をブロック
- `3` ローカル自動化: strict 型 + リンタールールが PostToolUse / pre-commit で自動実行
- `2` 手動運用: 型システムは有効、構造テストなし。コードレビューで人が構造を守っている
- `1` 形骸化: 型システムはあるが any が蔓延／ADR が古く設計決定が追えない
- `0` 未構成: 型チェックなし・構造的制約なし

### D. エントロピー管理

リポジトリの劣化を検出・修復する仕組み。エージェントは古い情報と新しい情報を区別できない。

| 確認項目 | 観点 |
|---------|------|
| ドキュメントの鮮度 | agent-docs の内容がコードの現状と一致しているか |
| デッドルールの検出 | 古い・矛盾するルールが残っていないか |
| 未使用コードの管理 | dead code、unused dependencies が放置されていないか |
| リポジトリ衛生 | `.gitignore` が適切か。不要ファイルがコミットされていないか |
| 定期クリーンアップ | 型エラー・Lint違反の定期スキャンが仕組み化されているか |

共通ラダーの適用例:
- `5` 継続改善と計測: 腐敗指標（リンク切れ件数・unused deps 件数等）を計測し定期クリーンアップPRを自動発行
- `4` 強制ゲート: CI でドキュメント鮮度・unused deps・dead code をブロック
- `3` ローカル自動化: pre-commit / PostToolUse でリンク切れ・dead code・unused deps を自動チェック（renovate / dependabot など外部スケジュール実行は2-3の中間）
- `2` 手動運用: 棚卸しを定期的に人が実行している
- `1` 形骸化: `.gitignore` 等はあるが陳腐化、古いルールやデッドコードが放置されている
- `0` 未構成: メンテナンスの仕組みが1件もない

### E. 計画と実行の分離

エージェントの行動を段階的に制御する仕組み。計画なしの実行はスコープクリープを招く。

| 確認項目 | 観点 |
|---------|------|
| Plan Mode活用 | 非自明なタスクで計画フェーズを経ているか |
| Skill整備 | 繰り返しワークフローがSkill化されているか |
| Subagent活用 | 独立タスクの並行実行・コンテキスト分離ができているか |
| レビューフロー | 実装前レビュー / コミット前レビュー / PR前レビューの構造 |
| タスク分解 | 大きなタスクが検証可能な小単位に分解されているか |

共通ラダーの適用例:
- `5` 継続改善と計測: Skill/Subagent の利用率・効果を計測し、フローを継続改善
- `4` 強制ゲート: PR テンプレートで plan ファイル必須、レビューチェックリストでタスク分解を強制
- `3` ローカル自動化: Plan Mode + Skill が日常的に使われ、スクリプトで起動が半自動化
- `2` 手動運用: Plan Mode は使うが Skill 化は限定的、人の判断で使い分け
- `1` 形骸化: Skill 定義はあるが使われていない／Plan Mode が実質スキップされている
- `0` 未構成: 計画フェーズなしに実装を開始している

### F. ガードレール

エージェントの暴走を防ぐ安全装置。信頼しつつ検証する（Trust but verify）の思想。

| 確認項目 | 観点 |
|---------|------|
| 権限制御 | `allowed-tools` で必要最小限のツールに制限しているか |
| mainブランチ保護 | 直接コミット防止（pre-commitフック、ブランチ保護ルール） |
| AI/人間経路の非対称設計 | AI 経路（PreToolUse / PostToolUse Hook）と人間経路（git pre-commit）が意図的に分離されているか。pre-commit が重すぎて `--no-verify` で逃げる動機を作っていないか |
| ループ検出 | 同一エラー再発・retry上限超過で中断する仕組みがあるか |
| 破壊的操作の防止 | `rm -rf`、`git push --force` 等へのガードがあるか |
| セキュリティ制約 | 認証情報のハードコード防止、`.env` のgitignore、secret scanning |
| 評価者の独立性 | コードを書くエージェントとレビューするエージェントが別 session / 別 context で動いているか |

共通ラダーの適用例:
- `5` 継続改善と計測: ループ検出ヒット率・secret 検出件数・誤 Approve 件数を計測しルールを改善
- `4` 強制ゲート: CI/プラットフォーム設定で main 保護・secret scan・権限制御を強制
- `3` ローカル自動化: pre-commit でループ検出・main 保護・secret パターンチェックが走る
- `2` 手動運用: main 保護の pre-commit は手動セットアップ、セキュリティチェックは人が実施
- `1` 形骸化: `.gitignore` はあるが secret scan なし／main 保護の opt-out が常用されている
- `0` 未構成: 保護機構が1件もない
