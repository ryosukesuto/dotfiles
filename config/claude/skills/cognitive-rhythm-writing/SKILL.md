# /cognitive-rhythm-writing - 長文解説の認知リズム調整

長文の事実性と論理構造を維持したまま、具体と意味づけの往復や問いの回収によって読み進めやすくする。面白さのために結論を隠したり、元情報にない場面・感情・迷いを作ったりしない。

## 適用判定

### 適用する

- 読み物として読ませたい技術解説、一般解説、記事
- `types/tech`、`types/docs`、解説寄りの `types/guides` の長文散文
- 内容は揃っているが、段落が同じ調子で続いて読みづらい文章
- `/explainer-doc` で生成する問題・解決・代替案の説明部分

### 適用しない

- PRタイトル・本文、コミットメッセージ、PRレビューコメント
- 議事録、ToDo、Daily、週次・月次レビュー、作業ログ
- 手順書、Runbook、リファレンス、チェックリスト、MOC
- ADRのDecision、Consequences、チェックリストなど、結論先出しが必要な意思決定部分
- 短文、箇条書き中心の文書、検索性や即時判断を優先する文書

対象外なら書き換えず、理由と優先するSkillを1〜2文で案内する。

- 既存のPR本文を整える: `/proofread`
- git diffからPR本文を作る: `/create-pr`
- PRをレビューする: `/review-pr`
- その他: 具体名が分かれば文書固有のSkillやテンプレート。不明なら名称を推測せず、そのまま「文書固有のSkillやテンプレート」と案内

## 実行手順

### 1. 正本と出力方法を確定する

文書固有のSkill、プロンプト、テンプレート、ユーザーの文体指定を先に読む。これらは本Skillより優先する。

- 元資料にない事実、因果、発言、感情、迷いを補わない
- ファイル編集が明示されていれば対象ファイルを編集し、それ以外は修正案を返す
- 修正案では指定された対象範囲全体を示し、保護対象は原文のまま含める。ユーザーが差分だけを求めた場合は変更箇所だけを示す
- 判断材料が不足している箇所は平坦なまま残す。リズムを作るための創作はしない

### 2. 残すべき情報を固定する

編集前に次を特定する。ここで固定した情報は、文章を軽くする目的で削らない。

- 読者が最初に必要とする範囲・前提・結論
- 正本へのリンク、日付、ステータス、未確定事項
- 判断を支えるデータ、コード、発言、具体例
- 文書固有テンプレートが要求する見出しと順序

### 3. 散文だけを診断する

次の順で確認する。

1. **話題**: その文は対象の状況を更新しているか、文書の進行だけを実況しているか
2. **具体と意味**: 抽象的な主張に、数値・記録・コード・判断結果のいずれかが対応しているか
3. **列挙の着地**: 箇条書きの後で、各項目が今回の判断や影響へどうつながるか分かるか
4. **問いの回収**: 冒頭や節内で立てた問いが、同じ節か直後の節で答えられているか
5. **段落の調子**: 同じ長さ・同じ構文・同じ距離感の段落が続いていないか

文書の範囲、正本、ステータスを示す文は、文書について述べていても機能があるため残す。

### 4. 最小限だけ書き換える

診断で問題が見つかった箇所だけを直す。

- 進行実況を削り、対象の事実や判断から段落を始める
- 抽象的な説明の直後に、既存の具体を対応づける
- 長い説明が続く箇所に、確定事項を固定する短い文を置く
- 箇条書きの内容を繰り返さず、判断や影響へ接続する一文を置く
- 問いを使う場合は対象から自然に生じる問いだけにし、答えを先送りしすぎない
- 冒頭ではリスク、意外な差、素朴な期待と実態のずれを使えるが、結論を意図的に隠さない

短文・長文・問いを機械的な周期で配置しない。変化させる材料がなければ元の文を残す。

### 5. 非散文を保護する

次は原則として書き換えない。

- frontmatter、表、コード、Mermaid、引用
- URL、内部リンク、識別子、数値
- テンプレート必須見出し、チェックリスト

非散文の内容に誤りが疑われる場合は、リズム調整と混ぜずに別の指摘として扱う。

### 6. 最終確認する

- [ ] 適用対象の文書種別か
- [ ] 必要な前提、範囲、正本、結論を削っていないか
- [ ] 元情報にない事実・感情・問いを作っていないか
- [ ] 問いを放置せず回収したか
- [ ] 抽象的な主張が既存の具体へ着地しているか
- [ ] 文体上の仕掛けを本文で説明していないか
- [ ] 文書固有のSkillとユーザーの文体指定を守っているか

## 併用順序

1. 文書固有のSkillやテンプレートで事実と構造を確定
2. 本Skillで長文散文のリズムだけを調整
3. `/proofread` でユーザーの文体へ最終調整

`/proofread` も本Skillで対象にした散文へ限定する。保護対象の非散文、必要な前提、具体との接続が変わった場合は、短さより内容の完全性を優先して戻す。

## Gotchas

- 「このドキュメントの位置づけ」のような文でも、範囲や正本を示していれば削除しない
- 読ませるために結論、修正依頼、操作手順を遅らせない
- 書き手の告白や逡巡は、元資料に存在しない限り追加しない
- 全ての節を問いで始めない。問いがなくても欠陥ではない
- 「緊張」「回収」「ここで短く止める」など、編集技法の名前を成果物へ書かない
- リズム改善と事実確認を同時に行わない。事実の誤りは先に直す
