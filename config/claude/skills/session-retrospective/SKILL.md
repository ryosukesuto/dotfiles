# session-retrospective - セッション振り返り

セッション内の自分の挙動を4観点で振り返り、改善提案をチャットに出す。提案先はCLAUDE.md / rules/ / feedback memory / skill定義のいずれか。ファイル保存はしない。

## なぜこのskillが必要か

エージェントは同じ失敗を繰り返しがち。明示的に振り返りフェーズを挟むことで、暗黙のミスをルール化して次セッションに引き継げる。「自己評価の限界」があるので完璧な検出は望まない。1セッション1-3個の改善が出ればOK。

## 実行手順

### 1. スコープ確認

セッションの開始から現時点までを振り返り対象とする。直近の数ターンだけ見ない。会話履歴を頭から辿る。

ユーザーが特定範囲（「直近のPR作業だけ」等）を指定した場合はそちらを優先。

### 2. 既存ルールの把握

以下を読んで「既に明文化されている内容」を把握する。重複提案を避けるため必須。

```bash
ls ~/.claude/CLAUDE.md ~/.claude/rules/ 2>/dev/null
ls ~/.claude/projects/-Users-s32943-gh-github-com-ryosukesuto-dotfiles/memory/ 2>/dev/null
```

- `~/.claude/CLAUDE.md` — グローバル基本原則
- `~/.claude/rules/*.md` — クイックリファレンス
- `~/.claude/projects/-Users-s32943-gh-github-com-ryosukesuto-dotfiles/memory/MEMORY.md` — auto memory索引
- プロジェクトルートの `CLAUDE.md` — リポジトリ固有

MEMORY.md の索引だけ読めば十分。個別memoryファイルは「該当しそう」と感じたものだけRead。

### 3. 4観点で検出

各観点について会話履歴をスキャンする。検出ゼロなら無理に出さず「該当なし」とする。

#### 観点1: ユーザーからの訂正・フィードバック

検出パターン:
- 否定語: 「いや」「違う」「そうじゃない」「やめて」「stop」「don't」「no,」「待って」
- 修正要求: 「〜にして」「〜じゃなくて〜」「むしろ」「逆」
- 暗黙のpushback: 質問の言い換え、同じ依頼の再提示

何を見るか:
- 訂正が「単発のミス」か「ルール化できる方針」か。後者だけ拾う
- 訂正の根拠（理由）が会話に出ているか。出ていれば feedback memory の `**Why:**` に使える

#### 観点2: 同じ質問・確認の繰り返し

検出パターン:
- AskUserQuestion を3回以上出している → 1度に聞けたか、デフォルト挙動を決められるか
- 同じファイル/コマンドのRead/lsを繰り返している → 結果を覚えていない
- 「〜してよいですか」を頻繁に聞く → 自律性レベルの設定漏れ

何を見るか:
- ユーザーの回答パターンが安定していれば、デフォルト化できる
- 確認内容が「破壊的操作」なら維持。「些末な選択」なら撤廃候補

#### 観点3: skillの使い損ね

検出パターン:
- skillのトリガーワード（descriptionに記載）が会話に出たのに起動していない
- skillが扱う領域の作業を、skillを使わず手動でやっている

何を見るか:
```bash
ls ~/.claude/skills/
```
- 各skillのSKILL.md先頭（descriptionとトリガー）を必要に応じて確認
- 「使うべきだった」と判明したskillとタイミングを具体的に指摘

#### 観点4: ツール使い分けの非効率

検出パターン:
- Bash 1コマンドで済むのにAgent委譲した
- 単純なgrepなのにExplore agentを呼んだ（逆も: 広範な調査をメインでやった）
- 独立な複数ツール呼び出しを直列にした（並列化漏れ）
- haiku向きの定型処理にsonnet/opus使った（逆も）

判断基準は `~/.claude/rules/model-selection.md` を参照。

### 3.5. 優先度・反映先の判定基準

優先度:
- 高: 破壊的操作（rm / push / DB変更等）に関わる、または既出ルール無視が2回以上連続発生したパターン、または資産活用の失敗（既存skillの使い損ね等）
- 中: トークン/コスト効率に関わる非効率、頻度依存で高に昇格しうるもの
- 低: 1回限りの軽微な事象、記録はするがルール化はしないもの

反映先の優先順位（上ほど優先）:
1. 既存ファイル（CLAUDE.md / 既存 rules/xxx.md / 既存 feedback memory）への追記 — 既出概念の補強
2. 既存 memory の description / trigger 強化 — 起動条件が弱くて発火していない場合
3. 新規 rules/xxx.md 作成 — 既存に該当カテゴリがない場合のみ
4. 新規 feedback memory 作成 — 個別の振る舞いルール

既出ルールが「存在するのに守られていない」場合は新規追加せず既存の trigger 強化を優先する。新規追加はルール肥大化の原因。

trigger 強化の具体策（採用順）:
1. 既存 memory / rules の description 冒頭に違反パターンを 1 行追記して語彙を増やす
2. CLAUDE.md の関連節（「作業アプローチ」「セキュリティ」等）から該当ルールへの参照を 1 行追加
3. 機械的に防げる場合は PreToolUse hook の追加を提案（`bin/claude-guard-*` パターン）

「既出ルール無視 2 回以上」のカウント方法: 同一ルールの違反だけでなく、別ルール違反でも「既出ルールに従えていない」というメタパターンとして合算してよい。同セッション内で 2 回以上発生したら優先度高。

### 4. 出力

以下フォーマットでチャットに出す。テンプレ厳守。

```
## セッション振り返り

### 観点1: ユーザーからの訂正・フィードバック

- 検出: {具体的なやりとりの引用、ターン番号や時系列でわかるように}
- 提案: {具体的な改善内容}
- 反映先: {CLAUDE.md / rules/xxx.md / feedback memory `xxx.md` / skill `yyy` 等}
- 優先度: 高/中/低

（複数検出した場合は `検出1` / `検出2` と並列のリストで書く。1観点に最大3件まで）

（検出ゼロの場合）
該当なし
（補足したい判定根拠があれば1行だけ添えてよい。例: 「2回Readはあるが別質問への対応なので非効率ではない」）

### 観点2: 同じ質問・確認の繰り返し
（同上）

### 観点3: skillの使い損ね
（同上）

### 観点4: ツール使い分けの非効率
（同上）

---

### 総括

1セッションで出てきた最重要1件をバッククォートで強調して再掲。ユーザーに「これだけは反映しよう」と判断してもらう材料にする。

全観点が「該当なし」の場合は、総括も「今回は検出ゼロ。改善提案なし」とだけ書いて締める。無理に総括内容を捻り出さない。
```

マークアップ規約: グローバル CLAUDE.md で太字記法（`**`）は禁止されているため、強調にはバッククォートまたはリスト構造を使う。

検出が複数ある観点は箇条書きで分ける。1観点に最大3件まで。優先度低を3件出すより、優先度高を1件出すほうが価値が高い。

### 5. ユーザー判断を仰ぐ

提案を出した後、ユーザーに「どれを反映する？」と聞かない。ユーザーが選んで指示してくる前提。skillの責務は提案までで、反映実行はユーザー or 別skill（auto memory書き込み等）に委ねる。

## Gotchas

- 自己評価バイアス: 自分の挙動を自分で評価するため、見落としが起きる。「うまくいった」と思っている箇所こそ要注意。会話履歴の文言を引用ベースで判定すること（記憶ではなく事実から）
- 過剰提案: 4観点全てで何か出そうとしない。検出ゼロは「該当なし」で良い。ノイズを混ぜると本当に重要な提案が埋もれる
- 既存ルールとの重複: `~/.claude/CLAUDE.md` / `rules/` / MEMORY.md を読まずに提案すると、既に明文化されている内容を再提案してしまう。必ずStep 2を実行する
- 抽象的な提案: 「丁寧にやる」「気をつける」のような提案は出さない。具体的なルール文・反映先ファイル・トリガー条件まで書く
- 訂正の単発性: 1回限りの訂正をルール化するとルールが膨らみすぎる。「2回以上同じ系統の訂正があった」「ユーザーが明示的に強い言葉で否定した」のいずれかを満たすものだけルール化候補にする
- Codexセカンドオピニオン: MVPでは入れない。提案精度に不満が出てきたら `codex-review` skillを組み合わせる拡張を検討
