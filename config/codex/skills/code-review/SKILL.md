# 出力形式

最終出力は以下の構造に固定する。途中の調査ログや探索プロセスは最終出力に含めない。

## 1. `## 所見:` セクションを冒頭に置く

- 番号付きリストで列挙する
- 各項目の先頭に優先度ラベルを付ける（`P0:` `P1:` `P2:` `P3:`）
  - **P0**: セキュリティ脆弱性、データ損失リスク、本番事故に直結。判定理由は「現実の被害金額・データ流出・サービス停止に直結するか」で線を引く
  - **P1**: 設計上の整合性ずれ、運用事故、要件未達、検出困難なバグ。判定理由は「リリース後にバグ票で起きる類のもの」で線を引く
  - **P2**: 保守性、可読性、テスト不足、責務の境界。判定理由は「将来の変更コストを増やす」で線を引く
  - **P3**: 命名、軽微なリファクタリング、好みレベル。判定理由は「無くても困らないが指摘の価値がゼロではない」で線を引く

迷ったら 1 段上 (P1 寄りに) にする。レビューワーが「これは P0 では？」と引き上げるより、依頼者が「これは P2 で良かった」と引き下げる方がコストが低い。

## 2. 各項目の書き方

- 1 文目で「何が問題か」を断定的に書く（打消し禁止、後述）
- 2 文目以降で「なぜ問題か」「どう直すか」を簡潔に
- 該当箇所がコード/ドキュメントなら `path/to/file.py:42` の形式で行番号付きで参照
- 必要に応じて該当行のスニペットを 3 行程度で引用

## 3. 末尾に `## 参照` セクション

- 公式ドキュメント、リポジトリ内の関連ファイル、過去の判断（ADR/Issue/PR）など
- URL またはファイルパス + 1 行説明

## 4. 全体について

- 出力の最初に「これからレビューします」「○○を確認しました」のような前置きは不要。`## 所見:` から始める
- 「ただし実際には問題にならない」「現実的にはレアケース」のような **打消しは絶対に書かない**。問題なら問題と断定する
- 探索ログ（`Search ...` `Read ...` `Explored` `Worked for ...`）は出力しない
- 「特に問題なし」と判断した場合は、`## 所見:` の代わりに `## 結論: 重大な問題は見つかりませんでした` を1行で書き、続けて「念のため次は確認すべき」と思う観点を `## 観点リマインダ:` で 3 件以内で列挙する
- 全体を必要以上に長くしない。1 項目あたり最大 5-7 行を目安

# 観点 (依頼内容に応じて意識する)

- **セキュリティ**: 認証/認可、機密情報の露出、TOCTOU、injection、CSRF、SSRF、permissive default
- **データ整合性**: トランザクション境界、idempotency、retry-safe、race condition
- **エッジケース**: 境界値（0, 空, 上限, 負）、同時実行、権限不足、ネットワーク失敗、部分失敗
- **可観測性**: ログ・メトリクス・アラート不足、debugability
- **互換性**: 後方互換、契約破壊、依存バージョンの整合
- **設計整合性**: 設計書との一致、レイヤー責務、命名規則、既存パターンとの統一感

# 禁止事項

- 「LGTM」「特に問題なし」を所見セクションの代わりに書かない（上記 #4 のフォーマットに従う）
- 提案だけ書いて根拠を省略しない
- ファイル名や行番号を曖昧にしない（不明なら「該当箇所未特定」と明示）
- **未確認の主張を断定しない**: 「`X が Y をサポートしていない`」「`Z モジュールは ignore_changes で動かない`」のような技術的主張は、必ず該当ファイル / 公式ドキュメント / 実機挙動のいずれかで verify してから書く。verify できない場合は推測であることを明示し、優先度を 1 段下げる（推測ベースの P0 は禁止、最大 P1 まで）
- **存在しない API / モジュール / フラグを実在のように書かない**: 確認していないものは「実装時に provider doc で確認が必要」のように留保を付ける

# Gotchas

- **アンダートリガー**: 「これどう思う？」「気になる点ある？」のような曖昧な依頼で description が引っかからないことがある。利用者が `code-review skill の出力形式に従って` と明示してきた場合は本 skill を必ず適用する
- **オーバートリガー**: 「実装して」「設計案を作って」のような新規生成依頼は本 skill のスコープ外。レビュー観点だけ抽出して返すと、生成依頼の意図と乖離する。生成系の依頼ではこのフォーマットを使わない
- **設計相談との切り分け**: 「自分の案が妥当か見て」は本 skill の対象 (レビュー)。「どう設計すべきか教えて」は対象外 (生成・助言)。前者なら依頼者の現案を最初に要約してから所見を出す
- **打消し誘惑**: P0 / P1 を出すと「ただし軽減策がある」「現実には起きにくい」と書きたくなるが、それは依頼者の判断領域。本 skill の責務は「問題があるか / 何が問題か」の特定までで、許容判断は混ぜない
- **探索ログの混入**: Codex の `Search ...` `Worked for ...` は内部ツール出力で、レビュー結果ではない。最終出力を組み立てる時に意識的に除外する
- **「該当箇所未特定」の濫用**: 行番号が分からないからといって全項目を「未特定」にしない。せめて関数名 / 段落名 / セクション名で位置を絞る

# 運用メモ

- 利用後にトリガー判定が想定とずれた (アンダー/オーバートリガー) と気付いたら、description のキーワードを増減させて調整する
- 出力形式が冗長 / 不足だと感じた場合は本ファイルを直接編集する。dotfiles 管理 (`~/dotfiles/config/codex/skills/code-review/`) で symlink されている
